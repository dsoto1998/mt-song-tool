# MTST Audio Conversion — Design Proposal
**Date:** March 17, 2026
**Scope:** Inline "Fix Format" conversion for non-conforming Bit Depth and Sample Rate in the Stem Check panel
**Source reference:** `AudioConverter/audio_resampler_gui.py`, `AudioAnalyzerService.swift`, `AudioAnalysisView.swift`

---

## Overview

MT Song Tool's Stem Check panel already identifies audio files that fail format validation — specifically files that are not **44.1 kHz** and/or not **16-bit**. These produce orange issue badges (`"48kHz"`, `"24-bit"`, etc.) on the affected rows. Today, QA staff must leave the app and use the standalone AudioConverter tool to fix those files before re-scanning.

This proposal describes how to fold AudioConverter's conversion logic directly into MTST so users can fix format-only issues without leaving the app — while keeping the scope intentionally narrow: **only Sample Rate and Bit Depth**. Stem name issues, silence, and corruption are never auto-fixable and remain unaffected.

---

## What AudioConverter Already Solves

`audio_resampler_gui.py` does the following that MTST needs:

| Capability | AudioConverter Implementation | MTST Equivalent Needed |
|---|---|---|
| Detect FFmpeg availability | `ffmpeg_exists()` → `subprocess.run([ffmpeg, "-version"])` | `Process`-based check in Swift |
| Invoke FFmpeg per file | `run_ffmpeg_convert(src, dst, sample_rate, channels, bit_depth)` | Swift `Process` equivalent |
| Map bit depth → PCM codec | `BIT_DEPTH_TO_PCM_FMT = {16: "pcm_s16le", ...}` | Same mapping, hardcoded to `pcm_s16le` for 16-bit |
| Thread conversion off main thread | `threading.Thread(target=run, daemon=True).start()` | `DispatchQueue.global` (already used in `AudioAnalyzerService`) |
| Output folder strategy | Sibling folder: `"16 bit 44.1 output"` next to source | Same sibling pattern, or overwrite-with-backup |

The core FFmpeg command AudioConverter builds for a 44.1 kHz / 16-bit WAV conversion is:

```
ffmpeg -hide_banner -loglevel error -y \
  -i <source.wav> \
  -ar 44100 \
  -c:a pcm_s16le \
  <output.wav>
```

That is the only command MTST needs. Channels are intentionally omitted — MTST does not validate or enforce channel count.

## Proposed MTST Integration

### 1. Where it lives

A new **"Fix Format"** button appears in `AudioAnalysisView` when:
- At least one result has a format-only issue (`"48kHz"`, `"96kHz"`, `"24-bit"`, `"32-bit"`, or a combination)
- No other blocking issues exist on those same files (the button is still available if a file has both a format issue and a stem name issue — conversion fixes what it can)

The button sits in the Stem Check card header, right of the Clear button. It is hidden when no format issues are present.

### 2. New service method on `AudioAnalyzerService`

```swift
// AudioAnalyzerService.swift additions

/// Files from the last scan that have at least one format issue (sample rate or bit depth).
var formatNonConformingFiles: [AudioFileResult] {
    results.filter { result in
        result.issues.contains(where: isFormatIssue)
    }
}

private func isFormatIssue(_ issue: String) -> Bool {
    // Matches: "48kHz", "96kHz", "192kHz", "24-bit", "32-bit", "48kHz 24-bit", etc.
    issue.hasSuffix("kHz") || issue.hasSuffix("-bit") || issue.contains("kHz")
}
```

### 3. Conversion logic (Swift `Process` replacing Python `subprocess`)

AudioConverter's `run_ffmpeg_convert()` translates directly to Swift using `Foundation.Process`:

```swift
/// Runs FFmpeg to convert a single file to 44.1 kHz / 16-bit WAV.
/// Returns (success: Bool, errorMessage: String).
/// Must be called off the main thread.
private static func runFFmpegConvert(
    ffmpeg: String,
    source: URL,
    destination: URL
) -> (Bool, String) {
    try? FileManager.default.createDirectory(
        at: destination.deletingLastPathComponent(),
        withIntermediateDirectories: true
    )
    let proc = Process()
    proc.executableURL = URL(fileURLWithPath: ffmpeg)
    proc.arguments = [
        "-hide_banner", "-loglevel", "error", "-y",
        "-i", source.path,
        "-ar", "44100",
        "-c:a", "pcm_s16le",
        destination.path
    ]
    let errPipe = Pipe()
    proc.standardError = errPipe
    do {
        try proc.run()
        proc.waitUntilExit()
    } catch {
        return (false, error.localizedDescription)
    }
    if proc.terminationStatus == 0 {
        return (true, "")
    }
    let errData = errPipe.fileHandleForReading.readDataToEndOfFile()
    let errMsg = String(data: errData, encoding: .utf8) ?? "Unknown FFmpeg error"
    return (false, errMsg.trimmingCharacters(in: .whitespacesAndNewlines))
}
```

This is a near-exact Swift port of `run_ffmpeg_convert()` from `audio_resampler_gui.py`, with channels omitted (MTST does not need to normalize channel count) and the codec hardcoded to `pcm_s16le` for 16-bit.

### 4. FFmpeg availability check

AudioConverter uses `ffmpeg_exists()`. In Swift:

```swift
static func ffmpegPath() -> String? {
    // 1. Check for a bundled copy next to the app binary (future-proofing)
    if let bundled = Bundle.main.url(forAuxiliaryExecutable: "ffmpeg"),
       FileManager.default.isExecutableFile(atPath: bundled.path) {
        return bundled.path
    }
    // 2. Fall back to system PATH via /usr/bin/env
    let proc = Process()
    proc.executableURL = URL(fileURLWithPath: "/usr/bin/env")
    proc.arguments = ["which", "ffmpeg"]
    let pipe = Pipe()
    proc.standardOutput = pipe
    try? proc.run()
    proc.waitUntilExit()
    let out = String(data: pipe.fileHandleForReading.readDataToEndOfFile(), encoding: .utf8)?
        .trimmingCharacters(in: .whitespacesAndNewlines)
    return (out?.isEmpty == false) ? out : nil
}
```

If `ffmpegPath()` returns nil, the "Fix Format" button shows a tooltip/alert: *"FFmpeg is required. Install via Homebrew: `brew install ffmpeg`"* — mirroring AudioConverter's error dialog.

### 5. Output folder strategy

AudioConverter creates a sibling folder named `"16 bit 44.1 output"` next to the source folder. MTST should use the same pattern to avoid overwriting originals:

```
/path/to/stems/              ← original source folder (scanned)
/path/to/stems_44.1_16bit/   ← generated output folder
```

After conversion, `AudioAnalyzerService.analyze(folder:)` is called automatically on the output folder, so the Stem Check panel refreshes in place showing the newly converted files.

The user does **not** need to configure an output path — the sibling folder is always auto-named and auto-scanned. This keeps the UX minimal: one button click, progress indicator, automatic re-scan.

## UI Changes

### `AudioAnalysisView` header (updated)

```
[ waveform ]  Stem Check  · 12 files, 3 issues         [Fix Format]  [Clear]
```

- **"Fix Format"** button: visible only when `!analyzer.formatNonConformingFiles.isEmpty`
  - Uses `CompactSecondaryButtonStyle` (matching the existing "Clear" button style) but with `.accent` color tint to signal action
  - Disabled and shows a spinner while conversion is running (`analyzer.isConverting == true`)
  - On click: shows a confirmation sheet — *"Convert 3 non-conforming files to 44.1 kHz / 16-bit? A new folder will be created next to your stems folder."* → **Convert** / Cancel

### New state on `AudioAnalyzerService`

```swift
@Published var isConverting = false
@Published var conversionProgress: (current: Int, total: Int) = (0, 0)
@Published var conversionErrors: [String] = []   // filenames that failed
```

### In-card progress (while converting)

The same slot used by `scanningView` can be reused:

```
[ spinner ]  Converting 2 of 3…
```

After conversion completes (or partially fails), the service immediately calls `analyze(folder: outputFolder)` which transitions to the normal scanning state, then to results.

### Post-conversion result handling

- If all conversions succeeded → the re-scan results replace the card contents normally
- If some files failed → a non-blocking inline warning appears above the results: *"2 files could not be converted (see log). Showing results for converted files."*
- The output folder path is shown as a small dim label in the card footer so users know where their files landed

## What Does NOT Change

| Concern | Decision |
|---|---|
| Stem name issues (`"Check Stem Name"`, `"Wrong Caps"`, `"Extra Space"`) | Never converted — user must rename manually |
| Silent files | Never touched by conversion |
| Corrupted files | Never touched by conversion |
| Channel count | Not enforced by MTST, not modified by conversion |
| `copyBlocked` logic | Unchanged — conversion triggers a re-scan which produces new `AudioFileResult` objects; `hasAudioIssues` resolves naturally if conversion succeeded |
| Original files | Never overwritten — output always goes to a new sibling folder |

## Dependency: FFmpeg

AudioConverter bundles FFmpeg via PyInstaller. For MTST (a native Swift app), the options are:

1. **Require system FFmpeg** (simplest) — document `brew install ffmpeg` as a one-time setup step. The `ffmpegPath()` check surfaces a clear message if it's missing.
2. **Bundle FFmpeg inside the `.app`** — include the FFmpeg binary in `Sources/MTSongTool/Resources/` and reference it via `Bundle.main.url(forAuxiliaryExecutable:)`. No user setup required, but adds ~70MB to the app bundle. This matches what AudioConverter does.

For an internal QA tool used by a small team, **option 1 (system FFmpeg via Homebrew) is the recommended starting point** — it avoids bundle size concerns and the team already works in a technical environment where Homebrew is likely present.

## Suggested Implementation Order

1. Add `ffmpegPath()` and `runFFmpegConvert()` to `AudioAnalyzerService` — these are pure utility functions, easily unit-testable in isolation
2. Add `convertAndRescan()` and the new `@Published` properties
3. Add "Fix Format" button to `AudioAnalysisView` header (hidden when no format issues)
4. Add confirmation sheet and converting progress view
5. Add post-conversion error summary and output folder label
6. Test with a folder of mixed-format stems (48kHz/24-bit files alongside conforming 44.1kHz/16-bit files) to verify only non-conforming files are converted and the re-scan picks up the output folder correctly
